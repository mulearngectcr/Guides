# Introduction

When you open a website, submit a form, upload a photo, or delete a message, your device communicates with a server.

This communication happens using a protocol called **HTTP**.

HTTP defines a set of actions, called **HTTP methods**, that tell the server what we want to do.

For example:

- Do we want to fetch some data?
- Do we want to create something new?
- Do we want to update existing information?
- Do we want to delete something?

These actions are represented using different HTTP methods.

The four most common methods are:

```text
GET
POST
PUT
DELETE
```

These map directly to CRUD operations.

---

# What Is HTTP?

HTTP stands for:

> HyperText Transfer Protocol

It is the language that browsers, mobile apps, and servers use to communicate.

Whenever you visit:

```text
https://google.com
```

your browser sends an HTTP request to Google's servers.

The server processes the request and sends back a response.

---

# Client-Server Communication

In web development, we usually have two parts:

---

## The Client

The client is the thing making the request.

Examples:

- Web browsers
- Mobile applications
- Postman
- Swagger UI
- Other backend services

---

## The Server

The server receives requests and sends responses.

Examples:

- FastAPI applications
- Django applications
- Express.js applications
- Spring Boot applications

---

# The Request-Response Cycle

The communication process looks like this:

```text
Client
   │
   │ GET /todos
   ▼
Server
   │
   │ Returns todos
   ▼
Client
```

This is called the **request-response cycle**.

---

# The GET Method

GET is used to **retrieve data**.

It never creates, modifies, or deletes anything.

---

## Real-World Example

Imagine opening Instagram.

The app requests:

```text
GET /posts
```

The server returns your feed.

Nothing is modified.

Data is simply read.

---

## Todo Examples

Get all todos:

```http
GET /todos
```

Response:

```json
[
  {
    "id": 1,
    "title": "Learn FastAPI",
    "checked": false,
    "priority": "medium"
  }
]
```

---

Get a specific todo:

```http
GET /todos/1
```

Response:

```json
{
  "id": 1,
  "title": "Learn FastAPI",
  "checked": false,
  "priority": "medium"
}
```

---

# Why GET Should Not Change Data

GET requests are meant for reading only.

This means:

Good:

```http
GET /todos
```

Bad:

```http
GET /deleteTodo
```

If someone refreshes the page, we do not want data to disappear accidentally.

That is why GET requests should never modify data.

---

# The POST Method

POST is used to **create new resources**.

Whenever we want to add something new, we use POST.

---

# Real-World Example

Suppose you upload a photo to Instagram.

The application sends:

```http
POST /posts
```

with information about the image.

The server creates a new post.

---

# Creating a Todo

Example:

```http
POST /todos
```

Request body:

```json
{
  "title": "Learn FastAPI",
  "checked": false,
  "priority": "medium"
}
```

The server creates a new todo item.

---

# Understanding Request Bodies

GET requests usually don't send data.

POST requests often include a **request body**.

The body contains information that the server needs.

Example:

```json
{
  "title": "Learn FastAPI"
}
```

The server uses this information to create new resources.

We will explore request bodies in much more detail in the next guide.

---

# The PUT Method

PUT is used to **update existing data**.

---

# Real-World Example

Suppose you edit your Instagram bio.

The application sends:

```http
PUT /profile
```

The server updates your information.

---

# Updating a Todo

Example:

```http
PUT /todos/1
```

Request body:

```json
{
  "title": "Learn FastAPI",
  "checked": true,
  "priority": "medium"
}
```

The server updates the todo with ID 1.

---

# Why Do We Need PUT?

Imagine using only POST:

```http
POST /todos
```

Would that create a new todo?

Or update an existing one?

The server cannot know.

Using separate HTTP methods makes APIs easier to understand.

---

# The DELETE Method

DELETE is used to remove resources.

---

# Real-World Example

Suppose you delete a message on WhatsApp.

The application sends something like:

```http
DELETE /messages/15
```

The server removes the message.

---

# Deleting a Todo

Example:

```http
DELETE /todos/1
```

The server removes the todo with ID 1.

---

# CRUD and HTTP Methods

The relationship becomes:

| CRUD Operation | HTTP Method |
| -------------- | ----------- |
| Create         | POST        |
| Read           | GET         |
| Update         | PUT         |
| Delete         | DELETE      |

This pattern is used almost everywhere.

---

---

# What About PATCH?

You might sometimes see:

```http
PATCH /todos/1
```

PATCH is used to update only part of a resource.

Example:

Instead of:

```json
{
  "title": "Learn FastAPI",
  "checked": true,
  "priority": "medium"
}
```

we could send:

```json
{
  "checked": true
}
```

Only one field changes.

For now, we will focus on PUT because it is easier to understand.

---

# REST API Conventions

Modern APIs follow certain conventions.

---

## Good Examples

```http
GET    /todos
GET    /todos/1
POST   /todos
PUT    /todos/1
DELETE /todos/1
```

These are clear and predictable.

---

## Bad Examples

```http
GET /getTodos
GET /createTodo
GET /updateTodo
GET /deleteTodo
```

These ignore HTTP conventions.

Developers generally avoid this style.

---

# Testing HTTP Methods in Swagger

FastAPI automatically generates Swagger documentation.

Visit:

```text
http://127.0.0.1:8000/docs
```

You will see different colors for different methods.

Typically:

```text

PUT    → Orange
DELETE → Red
```

This helps developers quickly identify the purpose of each endpoint.
