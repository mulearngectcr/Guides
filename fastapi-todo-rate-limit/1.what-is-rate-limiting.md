# 01 - What is Rate Limiting?

> Day 7 – Rate Limiting

---

# Introduction

Our Todo API currently has no restrictions on how many times someone can call it.

A single user could send thousands of requests per second.

This would cause serious problems.

Rate limiting solves this.

---

# What Is Rate Limiting?

Rate limiting means restricting how many times someone can call your API within a given time window.

For example:

```text
Maximum 10 requests per minute
```

If a user sends an 11th request within that same minute:

```text
429 Too Many Requests
```

The server rejects it.

After the minute resets, the user can make requests again.

---

# Real-World Example: OpenAI

When you use the OpenAI API, you have a limit on how many requests you can make per minute.

If you exceed it:

```json
{
  "error": {
    "message": "Rate limit reached",
    "type": "rate_limit_error"
  }
}
```

OpenAI uses rate limiting to ensure fair usage across all users.

---

# Real-World Example: GitHub API

GitHub allows:

```text
60 requests per hour for unauthenticated users
5000 requests per hour for authenticated users
```

If you exceed this limit, you receive:

```text
403 Forbidden
```

with a message explaining when your limit resets.

---

# Real-World Example: Google Maps

Google Maps API limits requests per day.

If you exceed your daily quota:

```text
429 Too Many Requests
```

You cannot make further requests until the next day.

---

# Why Does Rate Limiting Matter?

---

## Prevents Server Overload

Your server can only handle a certain number of requests at once.

Without rate limiting, one bad actor could send thousands of requests and crash your server for everyone.

---

## Prevents Abuse

Someone could scrape all your data in seconds.

Rate limiting forces them to slow down.

---

## Controls Costs

If your API calls paid services like an LLM, unlimited requests means unlimited bills.

Rate limiting caps your exposure.

---

## Ensures Fairness

One user sending too many requests should not slow things down for everyone else.

---

# What Is a Time Window?

A time window is the period over which requests are counted.

Examples:

```text
5/minute  → 5 requests allowed per minute
100/hour  → 100 requests allowed per hour
1000/day  → 1000 requests allowed per day
2/second  → 2 requests allowed per second
```

After the window resets, the counter starts again from zero.

---

# What Is 429 Too Many Requests?

429 is the HTTP status code for rate limiting.

It means:

> You have sent too many requests in too short a time.

The server is not broken.

The client simply needs to slow down.

---

# Global vs Per-Endpoint Limits

---

## Global Limit

One limit applies to all endpoints equally.

```text
All endpoints: 10/minute
```

Simple to set up.

Every endpoint gets the same treatment.

---

## Per-Endpoint Limit

Different endpoints get different limits.

```text
GET /todos  → 10/minute
POST /todos → 3/minute
```

More flexible.

Write operations are typically stricter than read operations.

---

# Why Are Write Operations Stricter?

Reading data is cheap.

Creating, updating, or deleting data is expensive.

It takes more server resources.

It can also have side effects.

So APIs typically allow fewer writes than reads per time window.
