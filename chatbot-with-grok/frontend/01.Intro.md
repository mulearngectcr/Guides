# Why Do We Need a Frontend?

Let's imagine our backend is already running.

```
http://localhost:8000
```

We can test it using Swagger.

```
POST /chat

{
    "message":"Hello"
}
```

The backend returns

```json
{
    "response":"Hi! How can I help you today?"
}
```

Technically,

our chatbot already works.

So why build a frontend?

Because normal users don't want to manually send JSON requests.

Imagine asking your friend to chat with an AI by opening Swagger, filling out JSON, and clicking **Execute** every time they wanted to ask a question.

That would be a terrible user experience.

Instead, we'd like something that looks like this.

```
┌──────────────────────────────────────┐
│ 🤖 AI Chatbot                        │
│                                      │
│ You: Hello                           │
│ Bot: Hi!                             │
│                                      │
│ You: What is Python?                 │
│ Bot: Python is...                    │
│                                      │
│ ──────────────────────────────────── │
│ Type your message...            [▶]  │
└──────────────────────────────────────┘
```

Much better.

The frontend transforms API calls into a natural conversation.

# What is Streamlit?

Streamlit is an open-source Python framework used for building web applications.

Unlike traditional frontend frameworks, Streamlit allows us to create user interfaces using only Python.

For example,

instead of writing

```
HTML

CSS

JavaScript
```

we simply write

```python
st.title("AI Chatbot")
```

or

```python
st.button("Send")
```

and Streamlit automatically generates the webpage.

This makes it an excellent choice for

- AI applications
- Data science projects
- Dashboards
- Internal tools
- Rapid prototypes

---

# Why Are We Using Streamlit?

There are several reasons.

## 1. We Already Know Python

Throughout this sprint, we've been writing Python.

Instead of learning an entirely new frontend language, we can continue using the same language.

---

## 2. Very Little Code

Creating a chat interface in React requires

- Components
- CSS
- State management
- Routing
- API calls

With Streamlit,

the same interface can be built in fewer than 50 lines of Python.

---

## 3. Built-in Chat Components

Streamlit already provides components like

```python
st.chat_message()
```

and

```python
st.chat_input()
```

These are specifically designed for chatbot applications.

We don't have to build the chat interface from scratch.

---

## 4. Great for AI Prototypes

Many AI startups use Streamlit during the early stages of development.

Instead of spending weeks designing a frontend,

they first build a working prototype.

Once the AI product is validated,

they may later replace Streamlit with React or another frontend framework.

This allows developers to focus on the AI rather than frontend complexity.

# Our Frontend Architecture

Our frontend is surprisingly simple.

```
              User
                │
                ▼
         Streamlit Interface
                │
      HTTP POST Request
                │
                ▼
        FastAPI Backend
                │
                ▼
           Groq API
                │
                ▼
        FastAPI Backend
                │
      JSON Response
                │
                ▼
         Streamlit Interface
                │
                ▼
              User
```

Notice something important.

The frontend **never communicates directly with Groq**.

It only communicates with our backend.

The backend handles everything else.

This keeps our API key secure and centralizes all AI-related logic in one place.

# Project Structure

Our project now looks like this.

```
chatbot/
│
├── backend.py
├── frontend.py
├── requirements.txt
├── .env
├── memory.json
└── .gitignore
```

Let's briefly understand each file from the frontend's perspective.

| File | Purpose |
|------|----------|
| `frontend.py` | Builds the user interface using Streamlit |
| `backend.py` | Processes user messages and communicates with the AI |
| `.env` | Stores configuration such as API URLs or API keys |
| `memory.json` | Stores conversation history on the backend |
| `requirements.txt` | Lists all project dependencies |
| `.gitignore` | Prevents unnecessary or sensitive files from being uploaded to GitHub |

Notice that the frontend never directly reads `memory.json`.

That file belongs entirely to the backend.

This separation of responsibilities makes the application easier to maintain.
