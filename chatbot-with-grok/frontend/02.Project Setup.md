# Frontend Guide
# Project Setup

Before we start building our chatbot interface, we need to set up our frontend environment.

In this chapter, we'll install the required libraries, understand why we need each one, and run our very first Streamlit application.

By the end of this chapter, you'll have a working Streamlit project ready to communicate with your FastAPI backend.

---

# What Does Our Frontend Need?

Our frontend has only one job.

```
User
    │
    ▼
Type a message
    │
    ▼
Send request to FastAPI
    │
    ▼
Receive AI response
    │
    ▼
Display response
```

To achieve this, we'll use two main libraries.

| Library | Purpose |
|----------|----------|
| Streamlit | Builds the user interface |
| Requests | Sends HTTP requests to the FastAPI backend |

We'll also use two helper libraries.

| Library | Purpose |
|----------|----------|
| python-dotenv | Loads environment variables from a `.env` file |
| os | Reads environment variables inside Python |

---

# Installing Streamlit

The first dependency we'll install is Streamlit.

Open your terminal and run

```bash
pip install streamlit
```

This downloads and installs the Streamlit framework along with all the packages it depends on.

Once the installation completes, your computer now has everything needed to create Streamlit web applications.

---

# Installing Requests

Our frontend needs to communicate with the backend using HTTP.

For this, we'll use the `requests` library.

Install it using

```bash
pip install requests
```

The `requests` library is one of the most widely used Python libraries for making HTTP requests.

Instead of manually creating network connections, sending headers, and handling responses, `requests` provides simple functions such as

```python
requests.get(...)
```

and

```python
requests.post(...)
```

We'll use `requests.post()` to send user messages to our chatbot backend.

---

# Installing python-dotenv

We'll also use environment variables to store our backend URL.

Install the package using

```bash
pip install python-dotenv
```

Although we could directly write the backend URL inside our code, using environment variables makes our application much easier to configure.

For example,

during development,

our backend URL might be

```
http://127.0.0.1:8000/chat
```

After deployment,

it could become

```
https://your-backend.onrender.com/chat
```

Instead of modifying the source code every time,

we only update the `.env` file.

---

# Updating requirements.txt

Whenever you install a new package, remember to update your `requirements.txt` file.

You can generate it automatically using

```bash
pip freeze > requirements.txt
```

This creates a file containing every package required by your project.

For example,

```
streamlit==1.xx.x
requests==2.xx.x
python-dotenv==1.xx.x
```

When someone else downloads your project—or when you deploy it—these packages can be installed automatically using

```bash
pip install -r requirements.txt
```

This ensures everyone uses the same dependencies.

# Creating the Project Structure

At this stage, your project should look something like this.

```
chatbot/
│
├── backend.py
├── frontend.py
├── .env
├── memory.json
├── requirements.txt
└── .gitignore
```

Let's briefly understand the new files.

### frontend.py

This is where we'll write our Streamlit application.

Everything related to the user interface will go here.

---

### .env

This file stores configuration values.

For now, we'll store the backend URL.

Example

```
API_URL=http://127.0.0.1:8000/chat
```

Later, after deployment, you'll simply replace this URL with your deployed backend URL.

---

### requirements.txt

Contains all Python dependencies.

Deployment platforms use this file to install the required packages automatically.

# Why Use Environment Variables?

Suppose we write

```python
API_URL = "http://127.0.0.1:8000/chat"
```

directly inside our code.

Everything works on our own computer.

Now imagine sharing the project with a friend.

Their backend might be running at

```
http://localhost:8001/chat
```

or

```
https://example.onrender.com/chat
```

They would need to edit the source code.

Instead,

we write

```
API_URL=...
```

inside the `.env` file.

The code remains unchanged.

Only the configuration changes.

This is a common software engineering practice because it separates **configuration** from **application logic**.

---

# Running a Streamlit Application

Unlike regular Python programs,

we don't start Streamlit using

```bash
python frontend.py
```

Instead,

we use

```bash
streamlit run frontend.py
```

Why?

Because Streamlit starts a web server behind the scenes.

Your Python script is responsible for describing the interface,

while Streamlit is responsible for displaying that interface inside a web browser.

When you run the command,

you'll see something similar to

```
You can now view your Streamlit app in your browser.

Local URL:
http://localhost:8501
```

Opening that URL displays your application.

---

# Why Not Use python frontend.py?

Suppose you run

```bash
python frontend.py
```

Python simply executes the file like any normal script.

There is no web server.

No webpage is created.

No user interface appears.

Streamlit applications require the Streamlit runtime, which is why we always use

```bash
streamlit run
```

instead of

```bash
python
```

# A Quick Test

Let's verify that everything is installed correctly.

Replace the contents of `frontend.py` with

```python
import streamlit as st

st.title("🤖 AI Chatbot")

st.write("Welcome to our chatbot!")
```

Now run

```bash
streamlit run frontend.py
```

If everything is configured correctly,

your browser should display a page containing

```
🤖 AI Chatbot

Welcome to our chatbot!
```

Congratulations!

You've just created your first Streamlit application.

Although it doesn't communicate with our backend yet, the frontend environment is now fully set up and ready for development.
