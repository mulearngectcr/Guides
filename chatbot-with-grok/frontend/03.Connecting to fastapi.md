# Frontend Guide

# Part 3 - Connecting the Frontend to FastAPI

Our chat interface is now complete.

Users can:

- view previous conversations,
- type new messages,
- and see their messages immediately.

The only missing piece is getting a response from our FastAPI backend.

This section completes our chatbot by sending the user's prompt to the backend and displaying the AI's reply.

---

# Creating the Assistant Message

After displaying the user's message, we create a chat bubble for the assistant.

```python
with st.chat_message("assistant"):
```

Everything inside this block will be displayed as the assistant's response.

Unlike the previous section, we don't have any response yet.

We'll generate it by calling our FastAPI API.

---

# Showing a Loading Indicator

AI responses usually take a few seconds to generate.

Instead of leaving the user wondering whether the application is working, we display a loading spinner.

```python
with st.spinner("Thinking..."):
```

When this block starts executing, Streamlit displays

```
Thinking...
```

along with a loading animation.

The spinner automatically disappears when the block finishes.

---

# Sending the Request

Inside the spinner, we send the user's message to our FastAPI backend.

```python
response = requests.post(
    API_URL,
    json={"message": prompt},
    timeout=60
)
```

Let's understand each argument.

### `API_URL`

This is the endpoint we loaded from the `.env` file.

```python
API_URL
```

It points to the `/chat` endpoint of our FastAPI backend.

### `json`

```python
json={"message": prompt}
```

This sends the request body as JSON.

For example, if the user types

```
What is FastAPI?
```

the frontend sends

```json
{
  "message": "What is FastAPI?"
}
```

This matches the request model we created in the backend.

### `timeout`

```python
timeout=60
```

This tells the `requests` library to wait for at most 60 seconds.

If the backend doesn't respond within that time, an exception is raised instead of waiting forever.

---

# Handling the Response

Once FastAPI processes the request, it sends a JSON response.

A successful response looks like this.

```json
{
  "response": "FastAPI is a modern Python web framework..."
}
```

To check whether the request succeeded, we write

```python
if response.status_code == 200:
```

A status code of **200** indicates that the request completed successfully.

If that's the case, we extract the AI's response.

```python
answer = response.json()["response"]
```

Here,

`response.json()` converts the JSON response into a Python dictionary.

Then,

```python
["response"]
```

retrieves the actual text generated by the AI.

---

# Handling Errors

Sometimes, the request may fail.

For example,

- the backend may not be running,
- the API may return an error,
- or the request may time out.

To prevent our application from crashing, we wrap the request inside a `try-except` block.

```python
try:
    ...
except Exception as e:
    answer = str(e)
```

If an exception occurs,

its error message is stored in the `answer` variable.

This allows us to display the error inside the chat interface instead of terminating the application.

We also handle API errors separately.

```python
else:
    answer = f"Error: {response.text}"
```

If FastAPI returns a status code other than `200`, we display the response body so the user knows something went wrong.

---

# Displaying the Assistant's Reply

Once we have the response,

displaying it is straightforward.

```python
st.markdown(answer)
```

Since we're already inside

```python
with st.chat_message("assistant"):
```

the text appears inside the assistant's chat bubble.

Whether `answer` contains an AI response or an error message, it is rendered in the same location.

---

# Saving the Assistant's Response

Finally, we store the assistant's reply.

```python
st.session_state.messages.append({
    "role": "assistant",
    "content": answer
})
```

Just like the user's messages,

the assistant's responses are saved in Session State.

This ensures that the complete conversation remains available even after Streamlit reruns the script.

For example, after one interaction, the conversation history might look like this.

```python
[
    {
        "role": "user",
        "content": "What is FastAPI?"
    },
    {
        "role": "assistant",
        "content": "FastAPI is a modern Python web framework..."
    }
]
```

On the next rerun,

the loop from the previous section reconstructs this conversation automatically.

---

# Complete Request Flow

The entire application now follows this sequence.

```
User Types a Message

        │

        ▼

Display User Message

        │

        ▼

Send POST Request

        │

        ▼

FastAPI Backend

        │

        ▼

LLM Generates Response

        │

        ▼

Return JSON Response

        │

        ▼

Display Assistant Message

        │

        ▼

Store Response in Session State
```

---

# Final Application Flow

By combining the backend and frontend, we've built a complete AI chatbot.

1. The user enters a message in the Streamlit interface.
2. The frontend stores and displays that message.
3. A POST request is sent to the FastAPI backend.
4. FastAPI forwards the request to the language model.
5. The generated response is returned as JSON.
6. The frontend displays the response.
7. Both messages are stored in Session State, allowing the conversation to persist throughout the session.
