# Frontend Guide
# Building the User Interface
## Session State

In the previous section, we learned that Streamlit reruns our entire script every time the user interacts with the page.

At first, this may sound strange.

Let's understand why this creates a problem.

---

# The Problem

Imagine our application looked like this.

```python
st.title("🤖 AI Chatbot")

prompt = st.chat_input("Type your message...")
```

The user opens the application.

```
🤖 AI Chatbot

_____________________
Type your message...
```

They type

```
Hello
```

and press **Enter**.

The script immediately reruns.

Now imagine the user types another message.

```
How are you?
```

The script reruns **again**.

How does the application remember that the user already sent

```
Hello
```

The answer is:

It doesn't.

Every rerun starts from the beginning.

Every normal Python variable is recreated.

---

# A Simple Example

Consider this program.

```python
count = 0

count += 1

st.write(count)
```

What do you expect to see?

Many beginners think

```
1

2

3

4
```

after refreshing.

Instead,

Streamlit always displays

```
1
```

Why?

Because every rerun starts with

```python
count = 0
```

The previous value is lost.

---

# Why Does This Happen?

Remember,

Streamlit executes your script like this.

```
Run Script

↓

Wait for user interaction

↓

Run Script Again

↓

Wait

↓

Run Again

↓

Wait

↓

Run Again
```

Each run is completely independent.

Normal variables only exist while the script is running.

Once the run finishes,

those variables disappear.

---

# We Need Memory

Our chatbot should remember previous messages.

Imagine this conversation.

```
You:
Hello

Bot:
Hi!

You:
How are you?
```

If the previous messages disappear after every rerun,

the interface would only display

```
You:
How are you?
```

That isn't much of a conversation.

We need a place where Streamlit can store information **between reruns**.

---

# Introducing Session State

Streamlit provides exactly this feature.

It's called

```
Session State
```

Session State stores values that survive multiple reruns.

Think of it as a small storage box attached to a user's browser session.

Every time the script restarts,

that storage box is still available.

---

# Accessing Session State

Streamlit provides

```python
st.session_state
```

This behaves very similarly to a Python dictionary.

For example,

we can write

```python
st.session_state["name"] = "Alice"
```

Later,

even after another rerun,

we can access

```python
st.session_state["name"]
```

and it still contains

```
Alice
```

---

# Creating Our Message List

Inside our chatbot,

we write

```python
if "messages" not in st.session_state:
    st.session_state.messages = []
```

Let's understand this carefully.

---

# Does "messages" Already Exist?

The first line checks

```python
"messages" not in st.session_state
```

Imagine this is the very first time someone opens the application.

Session State is empty.

```
{}
```

There is no

```
messages
```

key yet.

Therefore,

the condition becomes

```
True
```

---

# Creating an Empty Conversation

Now Streamlit executes

```python
st.session_state.messages = []
```

This creates

```
messages
```

inside Session State.

Now it looks like

```python
{
    "messages":[]
}
```

The empty list represents our conversation.

---

# What Happens During the Next Rerun?

Suppose the user sends

```
Hello
```

The script reruns.

Now

```
messages
```

already exists.

So

```python
if "messages" not in st.session_state:
```

becomes

```
False
```

The empty list is **not** created again.

Instead,

Streamlit continues using the existing conversation.

This prevents our messages from being erased.

---

# Why Use an if Statement?

Imagine we wrote

```python
st.session_state.messages = []
```

without checking first.

Every rerun would create a brand-new empty list.

The conversation would reset after every message.

That's exactly what we don't want.

The `if` statement ensures the list is created **only once**, when the user first opens the application.

---

# What Does Session State Look Like?

Initially,

```
{}
```

After initialization,

```python
{
    "messages":[]
}
```

After the first message,

```python
{
    "messages":[
        {
            "role":"user",
            "content":"Hello"
        }
    ]
}
```

After the AI responds,

```python
{
    "messages":[
        {
            "role":"user",
            "content":"Hello"
        },
        {
            "role":"assistant",
            "content":"Hi!"
        }
    ]
}
```

As the conversation continues,

this list keeps growing.

---

# Why Store Messages in a List?

A conversation naturally happens in order.

```
User

↓

Assistant

↓

User

↓

Assistant
```

Lists preserve order.

That's exactly what we need.

Each element of the list represents one message.

---

# Why Does Each Message Have role and content?

Notice that every message looks like this.

```python
{
    "role":"user",
    "content":"Hello"
}
```

or

```python
{
    "role":"assistant",
    "content":"Hi!"
}
```

The

```
role
```

tells Streamlit whether the message belongs to the user or the assistant.

The

```
content
```

contains the actual text.

Later,

when displaying messages,

Streamlit uses the role to decide whether the message should appear on the left or the right side of the chat interface.

---

# Frontend Memory vs Backend Memory

At this point,

you might notice something interesting.

Didn't we already create

```
memory.json
```

in the backend?

Why do we need another memory system?

The answer is that they serve completely different purposes.

| Frontend Session State | Backend memory.json |
|-------------------------|--------------------|
| Displays previous messages | Gives context to the AI |
| Exists only while the app is open | Persists even if the server restarts |
| Lives inside the browser session | Lives on the backend server |

The frontend stores messages so that the user can **see** the conversation.

The backend stores messages so that the AI can **understand** the conversation.

Although they contain similar data, they solve different problems.

---

# A Helpful Analogy

Imagine you're chatting with a friend.

The chat application on your phone displays the conversation.

That's similar to **Session State**.

Meanwhile,

the messaging company's servers also store the conversation.

That's similar to **memory.json**.

Both contain the same messages,

but they exist for different reasons.

In this section, you learned one of the most important concepts in Streamlit.

You now understand:

- Why Streamlit reruns the script after every interaction.
- Why normal Python variables are lost between reruns.
- What Session State is.
- How to create values inside Session State.
- Why we initialize `messages` only once.
- How our chatbot remembers previous messages during a session.
- The difference between frontend memory and backend memory.

With Session State in place, our chatbot can now remember the conversation while the application is running.
