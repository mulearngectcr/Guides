# Importing json

```python
import json
```

Python provides a built-in module called `json`.

JSON stands for **JavaScript Object Notation**.

Despite its name, JSON isn't exclusive to JavaScript.

Almost every programming language uses JSON for exchanging data.

For example,

```json
{
    "name": "Alice",
    "age": 20
}
```

This is a JSON object.

Inside Python, however, the same data looks like

```python
{
    "name": "Alice",
    "age": 20
}
```

Although they appear similar, one is a JSON string while the other is a Python dictionary.

The `json` module allows us to convert between these two formats.

We'll use it to store our conversation history inside a file called

```
memory.json
```

Without the `json` module, Python wouldn't know how to read or write JSON files.

---

# Importing os

```python
import os
```

The `os` module allows Python to interact with the operating system.

Some common tasks include

- Reading environment variables
- Creating folders
- Renaming files
- Deleting files
- Checking file paths

In our chatbot, we only use it for one purpose:

Reading the API key.

Later you'll see code like

```python
os.getenv("GROQ_API_KEY")
```

This asks the operating system,

> "Do you have an environment variable called GROQ_API_KEY?"

If yes,

Python returns its value.

Without the `os` module, we couldn't access environment variables.

---

# Importing load_dotenv

```python
from dotenv import load_dotenv
```

Earlier we created a file called

```
.env
```

Inside it we stored

```env
GROQ_API_KEY=your_api_key_here
```

Python does **not** automatically read this file.

Instead,

the `python-dotenv` package provides the function

```python
load_dotenv()
```

Its job is simple.

It opens the `.env` file and loads every variable into the operating system's environment.

After calling

```python
load_dotenv()
```

Python can finally access

```python
os.getenv("GROQ_API_KEY")
```

Think of it like this.

Before

```
.env
        ❌
Python can't see it.
```

After

```
load_dotenv()

↓

Python now knows about all variables inside .env
```

# Importing CORSMiddleware

```python
from fastapi.middleware.cors import CORSMiddleware
```

This import often confuses beginners.

Let's first understand the problem.

Imagine your frontend runs here

```
http://localhost:8501
```

Your backend runs here

```
http://localhost:8000
```

Even though both are running on your own computer,

the browser considers them two different origins.

Browsers contain an important security feature called

**CORS**

which stands for

Cross-Origin Resource Sharing.

Without permission,

the browser blocks requests from one origin to another.

That's why you'll often see errors like

```
Blocked by CORS policy
```

or

```
Failed to fetch
```

The backend itself isn't rejecting the request.

The browser is.

`CORSMiddleware` tells FastAPI

> "It's okay. I trust these frontends."

Later we'll configure exactly which websites are allowed to communicate with our API.

# Importing BaseModel

```python
from pydantic import BaseModel
```

Whenever the frontend sends data,

it usually arrives as JSON.

Example

```json
{
    "message": "Hello"
}
```

FastAPI needs a way to verify that this data is correct.

Pydantic provides exactly that.

We'll create

```python
class ChatRequest(BaseModel):
    message: str
```

Now FastAPI knows

- a request must contain a field called `message`
- it must be a string

If the frontend sends

```json
{
    "message": 123
}
```

FastAPI automatically returns an error before our code even runs.

This automatic validation is one of FastAPI's biggest strengths.

# Configuring CORS

Now we configure the middleware.

```python
app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],
    allow_credentials=False,
    allow_methods=["*"],
    allow_headers=["*"],
)
```

Let's understand each argument.

---

## CORSMiddleware

This tells FastAPI which middleware should be added.

Middleware is simply code that executes before every request reaches your endpoint.

Think of it like a security checkpoint.

Every request passes through it first.

---

## allow_origins

```python
allow_origins=["*"]
```

The asterisk means

> Allow requests from any website.

This is useful during development.

In production,

you'll usually replace it with something like

```python
allow_origins=[
    "https://mychatbot.streamlit.app"
]
```

so only your own frontend can access the backend.

---

## allow_credentials

```python
allow_credentials=False
```

Credentials include

- Cookies
- Login sessions
- Authorization information

Our chatbot doesn't use any of these.

So we keep it disabled.

---

## allow_methods

```python
allow_methods=["*"]
```

This allows

- GET
- POST
- PUT
- PATCH
- DELETE

and every other HTTP method.

Our chatbot only uses POST,

but allowing all methods keeps the configuration simple.

---

## allow_headers

```python
allow_headers=["*"]
```

HTTP requests contain headers.

Examples include

```
Authorization

Content-Type

Accept
```

This configuration allows all headers.

---

# Creating the Groq Client

Finally,

we connect our application to Groq.

```python
client = Groq(
    api_key=os.getenv("GROQ_API_KEY")
)
```

Let's break it down.

First,

```python
os.getenv("GROQ_API_KEY")
```

reads the API key from the environment.

That value is passed into

```python
Groq(...)
```

which creates a client object.

Think of the client as your application's connection to the Groq servers.

Every future AI request will go through this object.

Instead of repeatedly writing

```
https://api.groq.com
```

and handling authentication manually,

we'll simply write

```python
client.chat.completions.create(...)
```

The client remembers

- your API key
- authentication
- API endpoint
- connection details

making future requests much simpler.
