# Understanding How the Entire Chatbot Works

Congratulations!

At this point, our backend is complete.

We've written all the code required to power our AI chatbot.

However, writing code is only one part of software development.

A much more important skill is understanding **how all the pieces work together**.

In this chapter, we'll follow a single user message through the entire system—from the moment the user presses **Enter** until the AI response appears on the screen.

By the end of this chapter, you should be able to visualize exactly what happens inside your application during every conversation.

---

# The Big Picture

Our chatbot consists of four major components.

```
┌─────────────────────┐
│     Frontend        │
│     (Streamlit)     │
└──────────┬──────────┘
           │
           │ HTTP Request
           ▼
┌─────────────────────┐
│      FastAPI        │
│      Backend        │
└──────────┬──────────┘
           │
           │ API Call
           ▼
┌─────────────────────┐
│      Groq API       │
│  Llama 3.3 70B LLM  │
└──────────┬──────────┘
           │
           │ AI Response
           ▼
┌─────────────────────┐
│     Frontend        │
└─────────────────────┘
```

Each component has a specific responsibility.

| Component | Responsibility |
|-----------|----------------|
| Streamlit | Accepts user input and displays responses |
| FastAPI | Receives requests, manages memory, communicates with Groq |
| memory.json | Stores conversation history |
| Groq API | Generates intelligent responses |

Notice that **the frontend never communicates directly with the AI model**.

Every request passes through our backend first.

---

# Step 1 — The User Types a Message

Imagine the user types

```
What is FastAPI?
```

and presses **Enter**.

At this point,

nothing has reached the AI yet.

The message only exists inside the frontend.

```
User

↓

"What is FastAPI?"
```

---

# Step 2 — Streamlit Sends an HTTP Request

The frontend executes

```python
response = requests.post(
    API_URL,
    json={
        "message": prompt
    }
)
```

This creates an HTTP POST request.

The request body looks like this.

```json
{
    "message":"What is FastAPI?"
}
```

This JSON is sent across the internet (or your local network) to the FastAPI backend.

---

# Step 3 — FastAPI Receives the Request

The request arrives at

```python
@app.post("/chat")
```

FastAPI now performs several tasks automatically.

It

- reads the JSON body,
- validates it using Pydantic,
- converts it into a Python object.

After validation,

our endpoint receives

```python
request.message
```

which contains

```
What is FastAPI?
```

Notice how we never manually parse JSON.

FastAPI handles everything for us.

---

# Step 4 — Calling the Chat Function

The endpoint executes

```python
response = chat(request.message)
```

Now the chatbot logic begins.

Everything that follows happens inside the `chat()` function.

---

# Step 5 — Loading Previous Conversation

The first thing our chatbot does is

```python
history = load_memory()
```

Suppose

```
memory.json
```

contains

```json
[
    {
        "role":"user",
        "content":"Hello"
    },
    {
        "role":"assistant",
        "content":"Hi!"
    }
]
```

This entire conversation is loaded into memory.

Now our chatbot knows everything that has happened so far.

---

# Step 6 — Appending the New Message

Next we execute

```python
history.append(
    {
        "role":"user",
        "content":"What is FastAPI?"
    }
)
```

Our conversation now becomes

```text
User:
Hello

Assistant:
Hi!

User:
What is FastAPI?
```

The AI still hasn't been contacted.

We're simply preparing the conversation.

---

# Step 7 — Sending Everything to Groq

Now we call

```python
client.chat.completions.create(
    model="llama-3.3-70b-versatile",
    messages=history
)
```

This is the most important step.

Notice that we're **not** sending only

```
What is FastAPI?
```

Instead,

we send

```text
User:
Hello

Assistant:
Hi!

User:
What is FastAPI?
```

The AI receives the complete conversation.

That's why it appears to remember previous messages.

In reality,

it remembers nothing.

We're simply reminding it every time.

---

# Step 8 — The LLM Generates a Response

Groq receives our request.

Internally,

the model processes

- every previous message,
- the latest user question,
- the order of the conversation.

After reasoning,

it generates something like

```
FastAPI is a modern Python framework used for building APIs quickly and efficiently.
```

This response is sent back to our backend.

---

# Step 9 — Extracting the Response

The backend receives a large response object.

We only need the generated text.

```python
answer = response.choices[0].message.content
```

Now

```python
answer
```

contains

```
FastAPI is a modern Python framework...
```

---

# Step 10 — Updating Memory

Now we append

```python
history.append(
    {
        "role":"assistant",
        "content":answer
    }
)
```

Our conversation becomes

```text
User:
Hello

Assistant:
Hi!

User:
What is FastAPI?

Assistant:
FastAPI is a modern Python framework...
```

Notice that we're storing **both** sides of the conversation.

This ensures the next request has complete context.

---

# Step 11 — Saving the Conversation

Next we execute

```python
save_memory(history)
```

The updated conversation is written back into

```
memory.json
```

If the server stops and starts again,

this file still exists.

Our chatbot continues from where it left off.

---

# Step 12 — Returning the Response

The `chat()` function returns

```python
return answer
```

Control goes back to the API endpoint.

The endpoint responds with

```json
{
    "response":"FastAPI is a modern Python framework..."
}
```

This JSON is sent back to the frontend.

---

# Step 13 — Streamlit Displays the Response

The frontend receives

```json
{
    "response":"FastAPI is a modern Python framework..."
}
```

It extracts

```python
response.json()["response"]
```

and displays it inside the chat window.

The user finally sees

> **FastAPI is a modern Python framework used for building APIs quickly and efficiently.**

The conversation is complete.

# Why Use a Backend?

A common question is:

> Why not call the Groq API directly from the frontend?

There are several important reasons.

## 1. Protecting API Keys

Your Groq API key should never be exposed to users.

If the frontend communicated directly with Groq, anyone could inspect the application's code or network requests and obtain your API key.

By keeping the key on the backend, only your server has access to it.

---

## 2. Managing Conversation History

The frontend only sends the latest message.

The backend is responsible for loading previous conversations, updating memory, and sending the complete message history to the LLM.

This keeps the frontend simple and lightweight.

---

## 3. Adding Business Logic

The backend is the ideal place to implement additional features such as:

- User authentication
- Rate limiting
- Chat history management
- Logging
- Input validation
- Content moderation
- Database integration

Without a backend, implementing these features securely becomes much more difficult.

---

# Limitations of Our Current Design

Our chatbot works well for learning, but it has a few limitations.

## Shared Memory

Every user shares the same `memory.json` file.

If two people use the chatbot simultaneously, they'll see the same conversation history.

In a production application, each user would have their own conversation stored in a database.

---

## File-Based Storage

We're storing data in a JSON file.

While this is simple and easy to understand, it's not ideal for applications with many users.

Real-world applications typically use databases such as PostgreSQL or MongoDB.

---

## No Authentication

Anyone who knows the API URL can send requests to the chatbot.

Production systems usually require users to log in before accessing the service.
