# Talking to the LLM

In the previous chapter, we learned how to store and retrieve conversation history.

Our chatbot now has memory.

However, it still doesn't know how to generate responses.

In this chapter, we'll connect our application to Groq's Large Language Model (LLM) and understand exactly how a conversation with an AI model works.

By the end of this chapter, you'll understand:

- How the Groq client sends requests
- Why chat models expect a list of messages
- How previous conversation is passed to the model
- How the model generates a response
- How we extract the answer
- Why we save the assistant's response back into memory

---

# The chat() Function

Everything happens inside one function.

```python
def chat(prompt: str):

    history = load_memory()

    history.append(
        {
            "role": "user",
            "content": prompt,
        }
    )

    response = client.chat.completions.create(
        model="llama-3.3-70b-versatile",
        messages=history,
    )

    answer = response.choices[0].message.content

    history.append(
        {
            "role": "assistant",
            "content": answer,
        }
    )

    save_memory(history)

    return answer
```

Although this function is only a few lines long, it performs almost everything required to make a chatbot work.

Let's understand it one step at a time.

---

# Creating the Function

```python
def chat(prompt: str):
```

We're creating a function named

```
chat()
```

It accepts one parameter

```python
prompt
```

which represents the user's latest message.

Example

```
Hello

What is Python?

Tell me a joke.

Who won the FIFA World Cup?
```

Each time the frontend sends a request, this function receives exactly one message.

Notice the type annotation.

```python
prompt: str
```

This tells Python that `prompt` should be a string.

Although Python doesn't enforce this at runtime, type hints make the code easier to understand and help IDEs provide better suggestions.

---

# Step 1 — Loading Previous Conversation

The first line inside the function is

```python
history = load_memory()
```

Remember the previous chapter?

Our conversation history is stored inside

```
memory.json
```

Calling

```python
load_memory()
```

returns something like

```python
[
    {
        "role":"user",
        "content":"Hello"
    },
    {
        "role":"assistant",
        "content":"Hi!"
    }
]
```

This list is now stored inside the variable

```python
history
```

Think of `history` as the chatbot's notebook.

It contains everything that has happened so far.

---

# Why Load History First?

Imagine we skipped this step.

Our code would only know the latest message.

Suppose the conversation is

```
User:
My name is Alice.

Assistant:
Nice to meet you, Alice.
```

Now the user asks

```
What's my name?
```

If we don't load previous messages, the model only receives

```
What's my name?
```

There is no way for the AI to answer correctly.

Loading history ensures that the model receives the full conversation.

---

# Step 2 — Adding the User's New Message

Next we write

```python
history.append(
    {
        "role": "user",
        "content": prompt,
    }
)
```

Let's examine it carefully.

---

## What is append()?

`history` is a Python list.

Lists allow us to add new elements using

```python
append()
```

Example

```python
numbers = [1, 2, 3]

numbers.append(4)
```

Now

```python
[1,2,3,4]
```

The same idea applies here.

Instead of numbers,

our list contains conversation messages.

---

# The New Message

We're appending

```python
{
    "role":"user",
    "content":prompt
}
```

Suppose

```python
prompt
```

contains

```
What is Python?
```

After appending,

our history becomes

```python
[
    {
        "role":"user",
        "content":"Hello"
    },
    {
        "role":"assistant",
        "content":"Hi!"
    },
    {
        "role":"user",
        "content":"What is Python?"
    }
]
```

Notice something important.

We haven't contacted the AI yet.

We're simply preparing the conversation.

---

# Why Add the User Message Before Calling the API?

This is one of the most common beginner questions.

Why not call the API first?

Because the AI should see the latest message.

If we contacted the API first,

the conversation would end at

```
Hello

Hi!
```

The model would never see

```
What is Python?
```

Appending first ensures the latest message is included.

---

# Step 3 — Contacting the AI

Now comes the most important line.

```python
response = client.chat.completions.create(
    model="llama-3.3-70b-versatile",
    messages=history,
)
```

This single function sends an HTTP request to Groq's servers.

Although it looks like a normal Python function call, the SDK performs a lot of work behind the scenes.

It:

- connects to Groq,
- authenticates using your API key,
- converts Python objects into JSON,
- sends the request,
- waits for the response,
- converts the response back into Python objects.

All of that happens automatically.

---

# Understanding client

Earlier we created

```python
client = Groq(
    api_key=os.getenv("GROQ_API_KEY")
)
```

The client already knows

- your API key,
- where the Groq servers are,
- how to authenticate,
- how to send requests.

Whenever we need an AI response,

we simply use this client.

---

# Understanding chat

```python
client.chat
```

Groq provides different APIs.

One of them is the Chat API.

This API is designed specifically for conversations.

Unlike older text completion models,

chat models expect messages with roles.

That's why we send

```python
messages=[
    ...
]
```

instead of a single string.

---

# Understanding completions

```python
client.chat.completions
```

A completion is simply the model completing a conversation.

Imagine the conversation ends with

```
User:
What is Python?
```

The model's job is to complete it by generating

```
Assistant:
Python is a high-level programming language...
```

Hence the name

```
completion
```

---

# Understanding create()

```python
create(...)
```

This actually sends the request.

Everything before this line happens on your computer.

Everything after this line waits for Groq's servers.

---

# Understanding the Model

```python
model="llama-3.3-70b-versatile"
```

This tells Groq which AI model should answer.

Different models have different strengths.

Some are

- faster,
- cheaper,
- more accurate,
- better at reasoning,
- better at coding.

Here we're using

```
llama-3.3-70b-versatile
```

which is a general-purpose conversational model.

Changing this value changes the AI you're talking to.

---

# Understanding messages

This parameter is the heart of conversational AI.

```python
messages=history
```

Suppose

```python
history
```

contains

```python
[
    {
        "role":"user",
        "content":"Hello"
    },
    {
        "role":"assistant",
        "content":"Hi!"
    },
    {
        "role":"user",
        "content":"What is Python?"
    }
]
```

This entire list is sent to the model.

Notice something important.

We never send just

```
What is Python?
```

Instead,

we send the complete conversation.

That's what gives the illusion of memory.

The model doesn't remember anything.

We remind it of everything.

---

# What Does Groq Return?

After generating an answer,

Groq sends back a large JSON object.

It looks something like this (simplified).

```python
{
    "choices":[
        {
            "message":{
                "role":"assistant",
                "content":"Python is a high-level programming language."
            }
        }
    ]
}
```

There is much more information,

including

- token usage,
- finish reason,
- model information,

but for our chatbot,

we only care about the generated message.

---

# Extracting the Answer

We write

```python
answer = response.choices[0].message.content
```

Let's break it apart.

---

## response

The entire response object.

---

## choices

The API returns a list called

```
choices
```

A model can generate multiple possible responses.

We're only requesting one,

so the list contains a single element.

---

## [0]

The first response.

---

## message

The generated message object.

---

## content

Finally,

the actual text produced by the model.

Example

```
Python is a high-level programming language used for web development, AI, automation and much more.
```

This is the value we want.

---

# Step 4 — Saving the Assistant's Reply

Now we append

```python
history.append(
    {
        "role":"assistant",
        "content":answer
    }
)
```

Notice the difference.

Earlier,

the role was

```
user
```

Now it's

```
assistant
```

The conversation history becomes

```python
User:
Hello

Assistant:
Hi!

User:
What is Python?

Assistant:
Python is a high-level programming language...
```

This updated history will be used during the next request.

---

# Why Save the AI Response?

Imagine we didn't.

The next conversation would become

```
User:
Hello

User:
What is Python?

User:
Who created it?
```

The model would never see its own previous answers.

The conversation would quickly become confusing.

Saving both sides of the conversation keeps everything consistent.

---

# Step 5 — Writing the Updated History

Finally,

we save the updated conversation.

```python
save_memory(history)
```

This overwrites

```
memory.json
```

with the latest conversation.

The next time a request arrives,

our chatbot starts from this updated history.

---

# Step 6 — Returning the Response

The final line is

```python
return answer
```

Remember,

the frontend doesn't need the entire response object.

It only needs the generated text.

Returning

```python
answer
```

keeps our API clean and simple.

---

# The Complete Flow

Let's summarize everything that happens.

```
User enters a message
            │
            ▼
chat(prompt)
            │
            ▼
Load previous history
            │
            ▼
Append user's message
            │
            ▼
Send complete conversation to Groq
            │
            ▼
AI generates a response
            │
            ▼
Extract response text
            │
            ▼
Append assistant response
            │
            ▼
Save updated conversation
            │
            ▼
Return answer
```

Every time the user sends a message, this exact sequence repeats.

Although the model itself has no memory, continuously sending the full conversation allows it to behave as if it remembers everything discussed so far.
