Our chatbot still has one major problem.

It has **no memory**.

---

# What Happens Without Memory?

Suppose you ask the chatbot:

> Who is Lionel Messi?

The chatbot replies:

> Lionel Messi is an Argentine football player.

Now you ask:

> Which club does he play for?

Humans immediately understand that **he** refers to Lionel Messi.

But what about the AI?

If we only send the second message,

```
Which club does he play for?
```

the model has absolutely no idea who **he** is.

To the AI, every request looks like a completely new conversation.

This produces responses like

> Could you tell me who "he" refers to?

Clearly, that's not how a chatbot should behave.

---

# How Do LLMs Remember?

Here's something surprising.

Large Language Models **do not actually remember anything.**

Every API request is completely independent.

The model has no permanent memory of previous conversations.

Instead, we provide the memory ourselves.

Imagine a conversation like this.

```
User:
Hello

Assistant:
Hi!
```

Later the user asks

```
What is my first message?
```

Instead of sending only

```
What is my first message?
```

our backend sends

```
User:
Hello

Assistant:
Hi!

User:
What is my first message?
```

Notice what happened.

We included the previous conversation along with the latest message.

From the model's perspective,

it looks like one long conversation.

That's the secret behind chatbot memory.

# Understanding role

The role tells the AI who spoke the message.

There are three common roles.

```
user
```

A message written by the user.

Example

```json
{
    "role":"user",
    "content":"Hello"
}
```

---

```
assistant
```

A response generated by the AI.

Example

```json
{
    "role":"assistant",
    "content":"Hi!"
}
```

---

```
system
```

Special instructions for the model.

Example

```json
{
    "role":"system",
    "content":"You are a helpful coding assistant."

```

System messages are optional,

We'll keep our chatbot simple and use only

- user
- assistant

## Opening the File

```python
with open(MEMORY_FILE, "r") as f:
```

Let's break this apart.

### open()

### MEMORY_FILE

# Loading JSON

```python
return json.load(f)
```

Remember,

files store text.

Python needs to convert that text into a Python object.

Suppose the file contains

```json
[
    {
        "role":"user",
        "content":"Hello"
    }
]
```

After

```python
json.load(f)
```

Python now has

```python
[
    {
        "role":"user",
        "content":"Hello"
    }
]
```

as a normal Python list.

Now we can append new messages,

modify it,

---

# Why try...except?

Suppose

```
memory.json
```

doesn't exist yet.

Without

```python
try
```

Python crashes with

```
FileNotFoundError
```

Instead,

we catch the error.

```python
except FileNotFoundError:
    return []
```

This simply returns an empty conversation.

The chatbot starts with no previous messages.

Much better than crashing.

# Saving Memory

Now let's look at the opposite operation.

```python
def save_memory(memory):
    with open(MEMORY_FILE, "w") as f:
        json.dump(memory, f, indent=4)
```

Earlier,

we read the file.

Now we're writing it.

---

## The Parameter

```python
memory
```

This represents the entire conversation.

Example

```python
[
    {"role":"user","content":"Hello"},
    {"role":"assistant","content":"Hi!"}
]
```

Whatever is passed into this function gets saved.

---

## Opening in Write Mode

```python
"w"
```

means

Write.

Unlike

```
r
```

this mode replaces the file contents with the new data.

---

# json.dump()

```python
json.dump(memory, f, indent=4)
```

Earlier we used

```
json.load()
```

Now we use

```
json.dump()
```

Notice the difference.

```
load()

↓

File → Python
```

```
dump()

↓

Python → File
```

The parameter

```python
indent=4
```

simply makes the JSON nicely formatted.

Without it,

everything would appear on one line.

Readable files make debugging much easier.

---

# Why Save the Conversation?

Suppose our chatbot only kept memory in a variable.

The moment we stop the server,

everything disappears.

By saving to a file,

the conversation survives server restarts.

When the backend starts again,

it simply loads

```
memory.json
```

and continues from where it left off.
