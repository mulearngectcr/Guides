# 01 - Project Setup

Before we start writing the chatbot, let's prepare our development environment.

A good project setup makes development smoother, prevents dependency issues, and keeps sensitive information secure.

By the end of this chapter, you'll have a fully configured project ready for development.

---

# What We'll Build

Our project consists of two parts:

1. **Backend**
   - Built using FastAPI
   - Responsible for communicating with the AI model
   - Stores conversation history

2. **Frontend**
   - Built using Streamlit
   - Provides a simple chat interface for users

Although users only interact with the frontend, the actual intelligence of the chatbot lives inside the backend.

The communication flow looks like this:

```
User
  │
  ▼
Frontend (Streamlit)
  │
HTTP Request
  │
  ▼
Backend (FastAPI)
  │
Groq API
  │
AI Response
  │
  ▼
Frontend
```

# Creating a Virtual Environment

Python projects usually depend on external libraries.

Imagine one project requires:

```
FastAPI 0.110
```

while another requires

```
FastAPI 0.95
```

Installing everything globally can create version conflicts.

A virtual environment solves this problem.

It creates an isolated Python installation specifically for this project.

Each project can have its own dependencies without affecting other projects.

---

## Create the Environment

Windows

```bash
python -m venv venv
```

Linux / macOS

```bash
python3 -m venv venv
```

This creates a folder called

```
venv/
```

which contains

- Python interpreter
- pip
- installed packages
- activation scripts

---

# Activating the Environment

Windows (PowerShell)

```powershell
.\venv\Scripts\Activate
```

Windows (Command Prompt)

```cmd
venv\Scripts\activate
```

Linux / macOS

```bash
source venv/bin/activate
```

Once activated you'll usually see

```
(venv)
```

before your terminal prompt.

This indicates that Python commands are now using the virtual environment instead of your global installation.

# Installing Dependencies

Our chatbot relies on a few external libraries.

Install them using

```bash
pip install fastapi uvicorn groq python-dotenv
```

Let's understand what each package does.

---

## FastAPI

```
pip install fastapi
```

FastAPI is the web framework we'll use for building our backend.

It allows us to create HTTP endpoints such as

```
GET /
POST /chat
```

Whenever the frontend sends a request, FastAPI receives it, processes it and returns a response.

Think of FastAPI as the bridge between the user and our AI model.

---

## Uvicorn

```
pip install uvicorn
```

FastAPI is only the application.

It cannot run by itself.

Uvicorn is the server responsible for actually hosting our FastAPI application.

When you execute

```bash
uvicorn backend:app --reload
```

you're telling Uvicorn

> Start my FastAPI application and listen for incoming requests.

Without Uvicorn, the backend cannot receive requests from the frontend.

---

## Groq

```
pip install groq
```

This is the official Python SDK provided by Groq.

Instead of manually making HTTP requests to the Groq API, this library provides a clean Python interface.

Without it, we'd have to manually write code like

```
POST https://api.groq.com/...
Authorization: Bearer ...
```

The SDK handles all of that internally.

---

## python-dotenv

```
pip install python-dotenv
```

Applications often require secret information like

- API Keys
- Database Passwords
- Secret Tokens

Hardcoding them into source code is a terrible practice.

Instead, we store them inside a `.env` file.

The `python-dotenv` package loads these values into our application automatically.

---

# Creating requirements.txt

Once all packages are installed, create

```
requirements.txt
```

This file lists every dependency required by the project.

Example

```
fastapi
uvicorn
groq
python-dotenv
streamlit
requests
```

Whenever someone clones your project, they only need to run

```bash
pip install -r requirements.txt
```

to install everything.

---

# Creating the Environment File

Create

```
.env
```

Inside it write

```env
GROQ_API_KEY=your_api_key_here
```

This file stores private information.

Never upload it to GitHub.

---

# Getting a Groq API Key

Visit

```
https://console.groq.com
```

Create an account.

Navigate to

```
API Keys
```

Create a new key.

Copy it.

Paste it into your `.env` file.

Your backend will automatically load this key while starting.

---

# Why Not Hardcode the API Key?

Instead of writing

```python
client = Groq(
    api_key="gsk_abc123..."
)
```

we write

```python
client = Groq(
    api_key=os.getenv("GROQ_API_KEY")
)
```

Why?

Imagine uploading your project to GitHub.

If the API key is hardcoded,

everyone who visits your repository can use it.

Someone could consume your API quota or misuse your account.

Using environment variables keeps secrets outside the source code.

This is considered a standard industry practice.

---

# Creating memory.json

Create a file named

```
memory.json
```

Initially it should contain

```json
[]
```

This file stores our conversation history.

For example,

```json
[
    {
        "role":"user",
        "content":"Hello"
    },
    {
        "role":"assistant",
        "content":"Hi! How can I help you today?"
    }
]
```

Every new message will be appended to this file.

Whenever the chatbot receives a new question, it reads this file first.

This allows the model to remember previous messages instead of treating every request as a brand-new conversation.

---

# Creating .gitignore

Create

```
.gitignore
```

Add

```
venv/
.env
memory.json
__pycache__/
```

Let's understand why.

### venv/

The virtual environment can be recreated on any machine.

Uploading thousands of installed packages to GitHub is unnecessary.

---

### .env

Contains secret API keys.

Never upload it.

---

### memory.json

Stores user conversations.

This file changes constantly.

It doesn't belong in version control.

---

### __pycache__/

Python automatically creates this folder to speed up execution.

These files are generated automatically and don't need to be shared.
