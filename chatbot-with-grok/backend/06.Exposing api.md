

So far, we've built almost everything our chatbot needs.

Our backend can

- load environment variables,
- connect to Groq,
- remember previous conversations,
- generate responses using an LLM,
- and save the updated conversation.

However, there's still one problem.

The frontend has no way to communicate with it.

At the moment, the `chat()` function only exists inside our Python program.

```python
answer = chat("Hello")
```

This works only if another Python function calls it.

Our Streamlit application, however, runs as a completely separate application.

It needs a way to send messages to our backend.

This is exactly why APIs exist.

---

# What is an API?

API stands for

**Application Programming Interface**

An API is simply a way for two software applications to communicate.

Imagine ordering food online.

```
You
 │
 ▼
Waiter
 │
 ▼
Kitchen
```

You don't walk into the kitchen yourself.

Instead,

you tell the waiter what you want.

The waiter carries your request to the kitchen.

The kitchen prepares the food.

The waiter brings the food back.

Our chatbot works almost exactly the same way.

```
Frontend
     │
     ▼
FastAPI Endpoint
     │
     ▼
chat()
     │
     ▼
Groq API
     │
     ▼
FastAPI
     │
     ▼
Frontend
```

The FastAPI endpoint acts as the waiter between the frontend and the AI model.

# The Home Endpoint

Let's first create a very simple endpoint.

```python
@app.get("/")
def home():
    return {
        "message": "Chatbot API is running!"
    }
```

This endpoint doesn't generate AI responses.

Its only purpose is to verify that the server is running correctly.

If you visit

```
http://localhost:8000/
```

you'll receive

```json
{
    "message":"Chatbot API is running!"
}
```

Think of it as a health check.

Whenever you're unsure whether your backend is running,

opening this URL is usually the quickest way to verify it.

---

# Understanding the Decorator

The first line is

```python
@app.get("/")
```

This line is called a **decorator**.

A decorator tells FastAPI

> "Whenever someone sends a GET request to `/`, execute the function below."

Notice that the decorator itself doesn't contain any business logic.

It simply maps a URL to a Python function.

---

# Understanding the Route

The string

```python
"/"
```

represents the route.

This is also called an endpoint.

Examples

```
/
```

```
/chat
```

```
/login
```

```
/users
```

Every endpoint represents a different feature of your application.

---

# The Function

```python
def home():
```

This is an ordinary Python function.

The only difference is that FastAPI now calls it whenever the corresponding endpoint is accessed.

---

# Returning Data

The function returns

```python
{
    "message":"Chatbot API is running!"
}
```

Notice that we're returning a Python dictionary.

FastAPI automatically converts this into JSON.

We don't have to write

```python
json.dumps(...)
```

FastAPI handles the conversion automatically.

The browser receives

```json
{
    "message":"Chatbot API is running!"
}
```

---

# Why Create a Home Route?

Although it looks unnecessary,

this endpoint is extremely useful.

During deployment,

it's often the first endpoint we test.

If

```
/
```

works,

we know

- FastAPI started correctly.
- Deployment succeeded.
- Networking is working.

If it doesn't,

there's no point debugging the chatbot itself yet.

# Receiving User Messages

Now let's build the real endpoint.

Our frontend sends data like this.

```json
{
    "message":"Hello"
}
```

How does FastAPI know what this JSON looks like?

We define a request model.

```python
class ChatRequest(BaseModel):
    message: str
```

---

# Why Do We Need a Request Model?

Suppose someone sends

```json
{
    "message":"Hello"
}
```

Everything is fine.

But what if they send

```json
{
    "text":"Hello"
}
```

or

```json
{
    "message":123
}
```

or

```json
{}
```

Without validation,

our backend would probably crash.

Pydantic solves this problem.

---

# Understanding BaseModel

Earlier,

we imported

```python
from pydantic import BaseModel
```

Now we inherit from it.

```python
class ChatRequest(BaseModel):
```

This tells Pydantic

> "This class represents incoming JSON."

---

# Defining Fields

Inside the model,

we write

```python
message: str
```

This means

Our request **must** contain

```
message
```

and

its value must be a string.

Valid request

```json
{
    "message":"Hello"
}
```

Invalid request

```json
{
    "message":123
}
```

FastAPI automatically returns an error.

No additional code is required.

---

# Creating the Chat Endpoint

Now we create the endpoint.

```python
@app.post("/chat")
def chatbot(request: ChatRequest):

    response = chat(request.message)

    return {
        "response": response
    }
```

This is the endpoint our frontend communicates with.

Let's understand it line by line.

---

# POST /chat

```python
@app.post("/chat")
```

This tells FastAPI

Whenever someone sends

```
POST /chat
```

execute the function below.

Unlike the home endpoint,

this one expects incoming data.

---

# Function Parameter

```python
request: ChatRequest
```

This is where the magic happens.

Suppose the frontend sends

```json
{
    "message":"Tell me a joke"
}
```

FastAPI automatically converts this JSON into a Python object.

Now

```python
request.message
```

contains

```
Tell me a joke
```

Notice something important.

We never call

```python
json.loads()
```

We never manually parse the request.

FastAPI and Pydantic handle everything for us.

---

# Calling the Chat Function

Next,

we write

```python
response = chat(request.message)
```

Remember our previous chapter?

The `chat()` function

- loads memory,
- appends the new message,
- contacts Groq,
- receives the answer,
- saves memory,
- returns the generated response.

Instead of rewriting all of that code,

we simply call the function.

This keeps our application modular.

Every function has one clear responsibility.

---

# Returning the Response

Finally,

we return

```python
{
    "response": response
}
```

Suppose

```python
response
```

contains

```
Python is a programming language.
```

The frontend receives

```json
{
    "response":"Python is a programming language."
}
```

Notice that we're returning a dictionary.

FastAPI automatically converts it into JSON before sending it over the network.

# Why Separate the API from the Chat Logic?

You might wonder why we didn't write all the chatbot code directly inside the endpoint.

For example,

```python
@app.post("/chat")
def chatbot(request: ChatRequest):
    ...
```

could have contained all the memory handling and Groq API calls.

Instead,

we created a separate `chat()` function.

This separation has several advantages.

- The endpoint is short and easy to read.
- The chatbot logic can be reused elsewhere.
- The code becomes easier to test.
- Future improvements only require changes in one place.

This follows an important software engineering principle called **Separation of Concerns**, where each part of the program has a single, well-defined responsibility.
